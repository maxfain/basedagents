## Issues Found

### ❌ Rate limiting fires after 1 request (Regression)
**Severity: High**

Expected: 5 requests succeed per minute, 6th is rate-limited.
Actual: Request 1 returns `grade=A`, all subsequent requests (2–6) return `rate_limited` immediately.

This means the D1-based rate limiter is either:
- Counting the current request before the window check (off-by-one)
- The window isn't resetting correctly between test runs
- Limit is incorrectly set to 1 instead of 5

Recommend checking the D1 rate limit handler — likely a `>=` vs `>` comparison bug on the counter check.

---

### ⚠️ GitHub scan: OpenFang score=15 not 0
**Severity: Low**

Expected `score=0` for a known malicious repo. Got `score=15, grade=F`.  
The grade is correct (F), but if the spec says 0 for confirmed malicious repos, there may be a floor/penalty not fully applied. Functionally the grade is right — this is more of a spec alignment issue.

---

### ⚠️ PyPI provenance bonus not applied to `requests`
**Severity: Low**

`requests` is a well-established, provenance-verified package. `provenance=null` in the response suggests the provenance bonus isn't being detected or returned for PyPI packages. Score of 79 (B) is passing, but the expected provenance signal is missing — worth investigating whether the PyPI provenance check is wired up in scanner_version=2.

## All Clear

Core functionality is solid: API health nominal, auth security patch confirmed working, npm/GitHub/PyPI scanning all producing reasonable results, rescan + queue endpoints operational, agent search and reputation endpoints healthy, SDK at correct version, git history clean.

**One regression needs immediate attention:** rate limiting triggers after 1 request instead of 5 — this would break legitimate users hitting the scan API in any burst pattern. The other two items (OpenFang score floor, PyPI provenance null) are low-priority follow-ups.
